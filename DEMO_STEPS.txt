# GitHub Native Companion Dashboard - Hackathon Demo Script

This document provides a step-by-step guide to present the prototype to the judges.

## Prerequisites
1. Ensure both the Next.js frontend and FastAPI backend are running.
2. Have Postman or `curl` ready to mock GitHub webhook events.
3. Have the dashboard open in a browser.

## Demo Steps

### 1. Introduction (The "Remember" Phase)
- **Action:** Open the Dashboard UI (`http://localhost:3000`).
- **Talking Point:** Explain that the dashboard starts by digesting an entire GitHub repository to build its initial memory graph using Cognee.
- **Action:** Click the "Ingest Repository" button on the dashboard.
- **Visuals:** Show the "Repository Memory Status" updating to show that the initial structure, code, and documentation have been processed into the semantic graph.
- **Behind the Scenes:** Explain that the FastAPI backend called the GitHub API, fetched the default branch, and passed it to Cognee.

### 2. Push Webhook (The "Improve/Forget" Phase)
- **Talking Point:** Explain that the system stays up-to-date automatically via GitHub webhooks.
- **Action:** Send a simulated `push` webhook payload using `curl` or Postman to `http://localhost:8000/api/webhooks/github`.
  ```bash
  curl -X POST http://localhost:8000/api/webhooks/github \
       -H "Content-Type: application/json" \
       -H "X-GitHub-Event: push" \
       -d '{"ref": "refs/heads/main", "commits": [{"id": "abc1234", "added": ["new_feature.py"], "removed": ["old_code.py"]}]}'
  ```
- **Visuals:** Show the backend logs where Cognee runs `Improve` for the new files and `Forget` for the deleted ones.
- **Talking Point:** This ensures the memory graph is always a perfect representation of the current codebase.

### 3. Pull Request Review (The "Recall & Analyze" Phase)
- **Talking Point:** Now we show the core feature: the AI PR reviewer orchestrated by OpenClaw. OpenClaw routes the PR code changes, CI results, and architectural patterns recalled from Cognee to evaluate consistency.
- **Action (Trigger Accepted PR):** Send a simulated `pull_request` webhook payload with a title containing "Redis".
  ```bash
  curl -X POST http://localhost:8000/api/webhooks/github -H "Content-Type: application/json" -H "X-GitHub-Event: pull_request" -d "{\"action\": \"opened\", \"pull_request\": {\"number\": 44, \"title\": \"Implement Redis connection pool\"}}"
  ```
- **Visuals:** Navigate to the "PR Dashboard" -> **Accepted PRs** tab. Show how it parses successfully and conforms to the repository's established patterns.
- **Action (Trigger Rejected PR):** Send a simulated webhook payload with a title containing a conflicting system, e.g. "Memcached".
  ```bash
  curl -X POST http://localhost:8000/api/webhooks/github -H "Content-Type: application/json" -H "X-GitHub-Event: pull_request" -d "{\"action\": \"opened\", \"pull_request\": {\"number\": 45, \"title\": \"Add Memcached storage support\"}}"
  ```
- **Visuals:** Navigate to the "PR Dashboard" -> **Rejected PRs** tab. Show the AI reasoning explaining the architectural mismatch (i.e. conflicts with the Redis patterns retrieved from Cognee memory).
- **Behind the Scenes Visuals:** Show backend logs demonstrating:
  1. The API polling for CI completion.
  2. OpenClaw initiating the multi-agent review (Architecture & Code Quality agents).
  3. Cognee `Recall` operations fetching contextual memory based on the PR.

### 4. Conclusion
- Highlight that we avoided building another git hosting service. This is a purely native intelligence layer that lives alongside existing GitHub workflows.
